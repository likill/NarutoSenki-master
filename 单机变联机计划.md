## Codex评审与技术补充

### 结论

你的核心理解是正确的：当前项目没有实时联机，`NetworkLayer` 实际是竞技/HardCore 选人界面，真实战斗全部在 `GameLayer + ActionManager + Characters/Element` 本地运行。把它改成实时联机，最稳妥的方向确实是 Host 权威模型：Host 运行战斗、AI、伤害、刷兵、胜负，Client 只发送输入并渲染 Host 下发的结果。

但当前计划里有几处需要修正：

1. “新增9个文件”应改成“新增9个模块”。按 `.h/.cpp` 拆分后大约是 17 个源文件，还要改 `proj.win32/NarutoSenki.vcxproj` 和 `.filters`。
2. “不修改 ActionManager”不现实。可以不重构 8000+ 行主体，但至少要加少量网络钩子和实体 ID 字段，否则动态实体、伤害结果、技能特效无法可靠同步。
3. “只修改 LoadLayer.cpp 2 行和 StartMenu.cpp 一个按钮”过于乐观。至少还要改 `GameLayer.h/.cpp`、`LoadLayer.h/.cpp`、`ActionManager.h/.cpp`，以及工程文件。
4. “继承 GameLayer 拦截输入”目前不可直接成立。`HudLayer` 持有 `GameLayer* _delegate`，调用的是非虚函数 `JoyStickUpdate/attackButtonClick/attackButtonRelease/gearButtonClick`。如果只写 `HostGameLayer`/`ClientGameLayer` 同名方法，不会发生 C++ 虚派发。必须把这些入口改成 `virtual`，或者改成组合式输入路由。
5. Host-Client 更像局域网/房主开房，不等于《王者荣耀》式公网 dedicated server。公网移动网络会遇到 NAT、掉线、Host 作弊、Host 退出整局结束等问题。MVP 可以先做 LAN/同 Wi-Fi/端口转发；真正公网匹配要加中继或独立服务器。

### 推荐目标边界

第一版建议定义为：

- 2-6 人房间，空位由 Host AI 接管。
- Host 权威模拟，Client 不做伤害判定。
- Client 发送输入，Host 以固定频率广播快照和关键事件。
- 先支持 Win32 局域网；Android 之后再接。
- 先不做断线重连和反作弊；Host 退出则房间结束。
- 先不做完全帧同步/回滚，因为当前项目大量 `rand()`、Cocos Action、Scheduler、字符串状态和全局通知，不适合 lockstep。

### 需要改正的架构点

#### 1. GameLayer 输入入口必须可重写

`GameLayer.h` 中这些函数应改为 `virtual`：

```cpp
virtual void JoyStickUpdate(CCPoint direction);
virtual void JoyStickRelease();
virtual void attackButtonClick(abType type);
virtual void attackButtonRelease();
virtual void gearButtonClick(gearType type);
virtual void onGameStart(float dt);
virtual void addFlog(float dt);
virtual void onGameOver(bool isWin);
```

原因：`HudLayer` 通过 `GameLayer*` 调用这些方法，不加 `virtual` 时，`HostGameLayer/ClientGameLayer` 无法拦截。

同时建议给 `GameLayer` 增加通用方法：

```cpp
Hero* getHeroBySlot(int slotId);
Hero* getHeroByNetId(unsigned int netId);
void applyPlayerInput(int slotId, const NetInputFrame& input);
virtual bool isNetworkClient() const;
virtual bool isNetworkHost() const;
```

#### 2. role 不能继续承担“谁控制”的全部语义

现在 `role="Player"` 同时代表：

- 本地 HUD 目标。
- 本地输入目标。
- 某些技能、奥义、受伤、死亡 UI 分支。
- 非 AI 的玩家角色。

联机后需要新增 slot/owner 概念：

```cpp
// 建议加到 ActionManager 或 HeroElement
unsigned int _netId;       // 全局实体ID，Host分配
unsigned char _slotId;     // 0-5，英雄槽位；非英雄为255
unsigned char _ownerPeerId;// 控制者；AI为255
bool _isNetHuman;          // 这个实体由真人输入驱动
bool _isNetLocal;          // 当前客户端本地控制
```

MVP 可以保持每台机器只有自己的英雄是 `role="Player"`，其他真人英雄在本地表现为 `role="Com"` 但 `_isNetHuman=true`，并禁止 Host 对这些槽位调用 `doAI()`。更彻底的方案是把所有“本地 UI 判断”从 `role=="Player"` 改为 `isLocalPlayer(entity)`，但改动更大。

#### 3. Entity ID 是同步基础

现有 `_charNO` 只适合英雄/塔/minimap，不够表示刷兵、分身、傀儡、召唤物、子弹、陷阱。Host 必须给所有会影响战斗的实体分配 `netId`：

- 英雄：`1-8`
- 塔：`100-199`
- 小兵：`1000+`
- 召唤物/分身/陷阱/子弹：`20000+`

新增 `NetEntityRegistry`：

```cpp
class NetEntityRegistry {
public:
    unsigned int allocNetId();
    void bind(unsigned int netId, ActionManager* entity);
    ActionManager* find(unsigned int netId);
    void unbind(unsigned int netId);
};
```

注册点：

- `GameLayer::initHeros()`
- `GameLayer::initTower()`
- `GameLayer::initFlogs()/addFlog()`
- `ActionManager::setBullet()`
- `ActionManager::setClone()`
- `ActionManager::setMon()`
- `ActionManager::setTrap()`
- 实体 `dealloc()/dead()` 时解绑并广播 despawn

### 网络同步模型

#### 输入上行

Client 只上传输入，不上传位置/血量/伤害：

```cpp
struct NetInputFrame {
    unsigned int seq;
    unsigned int tick;
    unsigned char slotId;
    short dirX;       // -1000..1000
    short dirY;       // -1000..1000
    unsigned short buttonsDown; // 本帧按下
    unsigned short buttonsHeld; // 持续按住
    unsigned short buttonsUp;   // 本帧松开
    unsigned char gearType;
};
```

按钮位建议：

- bit0: `NAttack`
- bit1: `SKILL1`
- bit2: `SKILL2`
- bit3: `SKILL3`
- bit4: `OUGIS1`
- bit5: `OUGIS2`
- bit6: `Item1`
- bit7: `GearItem`

Host 每帧或每个网络 tick 处理输入：

```cpp
void GameLayer::applyPlayerInput(int slotId, const NetInputFrame& input) {
    Hero* hero = getHeroBySlot(slotId);
    if (!hero || hero->getActionState() == ACTION_STATE_DEAD) return;

    CCPoint dir = ccp(input.dirX / 1000.0f, input.dirY / 1000.0f);
    if (fabs(dir.x) > 0.01f || fabs(dir.y) > 0.01f) {
        hero->walk(ccpNormalize(dir));
    } else if (hero->getActionState() == ACTION_STATE_WALK) {
        hero->idle();
    }

    if (input.buttonsDown & BTN_SKILL1) hero->attack(SKILL1);
    if (input.buttonsDown & BTN_SKILL2) hero->attack(SKILL2);
    if (input.buttonsDown & BTN_SKILL3) hero->attack(SKILL3);
    if (input.buttonsDown & BTN_OUGIS1) hero->attack(OUGIS1);
    if (input.buttonsDown & BTN_OUGIS2) hero->attack(OUGIS2);
    if (input.buttonsDown & BTN_ITEM1)  hero->setItem(Item1);
    if (input.buttonsDown & BTN_NATTACK) hero->attack(NAttack);
}
```

注意：当前 `_isAttackButtonRelease` 是 `GameLayer` 全局字段，只适合单个本地玩家。多人时建议改成每个 slot 一个 `bool attackReleased[8]`，或者 MVP 先把远端普通攻击当作单次点击处理。

#### Host 下行快照

Host 每 10-15Hz 广播状态快照，Client 用插值/纠偏渲染：

```cpp
struct NetEntityState {
    unsigned int netId;
    unsigned char entityType;  // Hero/Flog/Tower/Monster/Bullet
    unsigned char slotId;
    unsigned char group;
    unsigned char role;
    int x;
    int y;
    int hp;
    int maxHp;
    int ckr;
    int ckr2;
    unsigned char level;
    unsigned char actionState;
    unsigned char flags; // flip/visible/wudi/bati/controlled/dead
};
```

快照只同步“结果状态”，不要求 Client 本地算出同样结果。这样可以绕开当前项目里的 `rand()`、Cocos Action 时间误差、AI 调度差异。

#### Host 下行事件

只靠快照会缺少技能表现，因此还要同步事件：

- `SpawnEntity`：刷兵、子弹、陷阱、召唤物、分身。
- `DespawnEntity`：实体移除。
- `ActionStart`：某实体开始 `NAttack/SKILL/OUGIS/walk/idle/hurt/dead`。
- `DamageApplied`：目标、来源、数值、hitType、是否暴击。
- `BuffApplied/BuffRemoved`
- `Transform`：Naruto -> SageNaruto、Sasuke -> ImmortalSasuke 等。
- `TowerDestroyed`
- `KillReport`
- `GameOver`
- `OugisCutin`

因此 `ActionManager` 至少需要以下轻量钩子：

```cpp
// setAttackBox() 里，Host 广播攻击判定事件；Client 只表现，不做权威伤害
_delegate->onNetAttackBox(this, _effectType, _attackValue, _attackRangeX, _attackRangeY);

// setDamage() 结束后，Host 广播真实伤害结果
_delegate->onNetDamageApplied(attacker, this, realValue, effectType);

// setBullet/setClone/setMon/setTrap 创建实体后，Host 注册并广播 SpawnEntity
_delegate->onNetEntityCreated(entity);
```

这不是重构 `ActionManager`，只是加出口。没有这些出口，Client 很难知道“哪个动态实体被创建、谁打了谁、真实伤害是多少”。

### 协议建议

第一版可以使用 TCP 长连接，便于调试；后续再升级 UDP/KCP/ENet。TCP 要加 `TCP_NODELAY`，避免小包延迟。

消息头：

```cpp
struct NetHeader {
    unsigned short magic;  // 0x4E53, "NS"
    unsigned char version;
    unsigned char type;
    unsigned int seq;
    unsigned int tick;
    unsigned short length;
};
```

消息类型：

```cpp
enum NetMsgType {
    MSG_HELLO = 1,
    MSG_HELLO_ACK,
    MSG_PING,
    MSG_PONG,

    MSG_ROOM_CREATE,
    MSG_ROOM_JOIN,
    MSG_ROOM_STATE,
    MSG_SLOT_UPDATE,
    MSG_READY,
    MSG_START_GAME,

    MSG_INPUT,
    MSG_SNAPSHOT,
    MSG_EVENT,
    MSG_ACK,
    MSG_DISCONNECT
};
```

握手必须校验：

- `protocolVersion`
- `CURRENT_VERSION`
- 房间人数和 slot
- 地图编号 `randomMap`
- 阵营 `zhenying`
- 选人表
- 资源/角色数据版本，至少先用 `CURRENT_VERSION`，后续再加资源 MD5

### 线程模型

不要在网络线程直接操作 Cocos 节点。推荐：

1. `NetManager` 线程负责阻塞 `recv/send`。
2. 收到包后解析成 `NetMessage`，推入线程安全队列。
3. `HostGameLayer/ClientGameLayer` 在主线程 `schedule(updateNetwork, 0.0f)` 中取队列并应用到游戏对象。
4. 发送包可以从主线程组包后交给 `NetManager` 发送队列。

Win32/Android 跨平台 Socket：

- Win32: `winsock2.h`, `WSAStartup`, `closesocket`
- Android/Linux: `sys/socket.h`, `arpa/inet.h`, `unistd.h`
- 所有多字节数字统一网络字节序，或者自定义小端写入并明确协议只走同端实现
- VS2010 不要依赖 C++11 线程/lambda，优先用 pthread 或平台线程

### GameLayer 初始化调整

`GameLayer::initHeros()` 当前默认 `i==0` 是本地 `currentPlayer`。联机应改为：

```cpp
bool isLocalSlot = (slotId == _localSlotId);
```

`onGameStart()` 中不要简单地对所有 `role=="Com"` 调 `doAI()`，应改成：

```cpp
if (isNetworkClient()) {
```

### ClientGameLayer 行为

Client 侧建议：

- 本地摇杆和按钮只打包 `MSG_INPUT` 发给 Host。
- 可以对本地英雄做移动预测，但不要本地结算伤害。
- 收到 `MSG_SNAPSHOT` 后按 `netId` 修正位置、血量、状态。
- 收到 `MSG_EVENT` 后播放技能、伤害数字、击杀、塔毁、奥义等表现。
- Client 不调用 `doAI()`，不主动 `addFlog()`，不主动随机地图。

为了防止 Client 本地 `attack()` 触发 `acceptAttack` 造成伪伤害，可加：

```cpp
bool GameLayer::isAuthoritativeCombat() const {
    return !isNetworkClient();
}
```

然后在 `ActionManager::setAttackBox()` 里：

```cpp
if (_delegate->isAuthoritativeCombat()) {
    CCNotificationCenter::sharedNotificationCenter()->postNotification("acceptAttack", this);
}
```

Client 如需播放攻击动画，可以播放动作，但不执行本地命中结算。

### 分阶段实现建议

#### Phase 0：铺底重构，1-2天

- 给 `GameLayer` 输入和生命周期入口加 `virtual`。
- 增加 `slotId/netId/localSlotId/roomData` 字段。
- 增加 `getHeroBySlot/applyPlayerInput`。
- 增加 `NetEntityRegistry`。
- 保证原单机模式不变。

#### Phase 1：房间和启动同步，2-4天

- TCP 连接、握手、ping/pong。
- Host 创建房间，Client 输入 IP 加入。
- 同步 slot/hero/ready/start。
- 所有人加载同一个 `tempHeros`、同一个地图、同一个阵营。

#### Phase 2：输入驱动 Host，3-5天

- Client 发送 `MSG_INPUT`。
- Host 把远端输入应用到对应 hero。
- Host 对 AI 槽调用 `doAI()`，真人槽不调用 AI。
- Host 本地也走同一套 `applyPlayerInput`，不要走两套逻辑。

#### Phase 3：快照同步，4-7天

- Host 定时广播英雄、塔、小兵状态。
- Client 建立 `netId -> entity` 映射。
- Client 根据快照纠正位置、血量、等级、状态。
- 先做到“能看见同一局结果一致”。

#### Phase 4：事件同步，5-10天

- 同步刷兵、子弹、召唤物、分身、陷阱。
- 同步伤害数值、死亡、重生、塔毁、击杀播报。
- 同步变身、奥义 cut-in、buff。
- 补齐高风险角色：Naruto/Sasuke/Kankuro/Chiyo/Kakuzu/Minato/Ino/Hidan。

#### Phase 5：手感和稳定性，持续迭代

- 本地移动预测 + 插值。
- 丢包/延迟 UI。
- Host 断开处理。
- 版本不一致提示。
- 日志和回放包保存，便于复现同步 bug。

### 时间评估修正

原估计 7-12 天适合“只做一个非常粗糙的 LAN 原型”：能进房、能开始、能看到远端人物动。

如果目标是 6 人、空位 AI、技能/召唤/伤害/塔毁基本正确，建议按 3-6 周估算。若还要公网联机、断线重连、移动端稳定、所有角色技能完全同步，按 2 个月以上更真实。

### 最小可验证里程碑

这个版本完成后，再扩展更多玩家、动态召唤、预测和公网能力。
