## PROJECT OVERVIEW

**NarutoSenki** is a 2D MOBA (Multiplayer Online Battle Arena) game built on **Cocos2d-x 2.2.2** using C++. It is a Naruto-themed side-scrolling battle game where two teams (Konoha vs Akatsuki) fight to destroy each other's towers. The game is single-player with AI-controlled teammates and enemies -- there is no real-time multiplayer networking.

## 4. GAME LOOP AND GAME FLOW

**Scene flow:** `GameScene -> StartMenu -> SelectLayer/NetworkLayer -> LoadLayer -> GameLayer (with BGLayer + HudLayer)`

**Game initialization sequence in GameLayer:**

1. `initHeros()` -- Creates the TMX tile map, spawns all heroes at spawn points from map object data, creates hero instances, assigns Player/Com roles, sets HP bars and shadows, calls `idle()` on each
2. `initTower()` -- Reads tower positions from TMX "meta" object group, creates Tower instances
3. Schedules `updateViewPoint` (0.0s interval = every frame) -- camera follows player
4. `onKaichang()` -- Shows "battle start" animation after 0.5s
5. `onGameStart()` -- After 0.75s more: initializes HUD, starts game clock, starts creep spawning, starts AI for all Com heroes, starts background music

**Main update loops (scheduled timers):**

- `updateViewPoint(float dt)` -- Every frame: camera follows `currentPlayer` position, clamped to map bounds
- `updateGameTime(float dt)` -- Every 1s: increments game timer, updates HUD clock
- `addFlog(float dt)` -- Every 15s: spawns 6 new creeps per team
- `checkBackgroundMusic(float dt)` -- Every 2s: checks/loops BGM
- `ActionManager::update(float dt)` -- Every frame per entity: updates buff effects, HP bar position, shadow position, handles base healing zone, handles walk movement (`_desiredPosition = position + velocity * dt`)

## 5. CHARACTER/HERO SYSTEM

**Identity:** Each entity has three string identifiers:

- `_character` -- Character name (e.g., "Naruto", "Sasuke", "Kakashi")
- `_role` -- Role: "Player", "Com", "Flog", "Tower", "Clone", "Summon", "Bullet"
- `_group` -- Team: "Konoha" or "Akatsuki"

**Character data** is loaded from XML files in `Resources/Element/[CharacterName]/[CharacterName].xml` via `KTools::readXMLToArray()`. Each character has:

- Idle, walk, hurt, dead, attack, and multiple skill animation frames
- Normal attack + 3 skills + 2 ultimate (Ougis) attacks
- Each attack has: type, value, range X/Y, cooldown, combat point cost

**60+ playable characters** including: Naruto, SageNaruto, RikudoNaruto, Sasuke, ImmortalSasuke, Kakashi, Sakura, Hinata, Neji, Lee, RockLee, Shikamaru, Gaara, Jiraiya, SageJiraiya, Tsunade, Itachi, Deidara, Konan, Kisame, Pain, Hidan, Kakuzu, Tobi, Orochimaru, Minato, Tenten, Sai, Asuma, Choji, Ino, Kiba, Shino, Kankuro, Chiyo, Suigetsu, Karin, Jugo, Hiruzen, Nagato, Tobirama, and more.

**Character transformations:** Some characters transform mid-game (e.g., Naruto -> SageNaruto -> RikudoNaruto, Jiraiya -> SageJiraiya, Sasuke -> ImmortalSasuke, RockLee <-> Lee).

## 6. MOVEMENT AND COMBAT (ActionManager)

**Movement:**

- JoyStick generates a `_direction` CCPoint
- `GameLayer::JoyStickUpdate(direction)` calls `currentPlayer->walk(direction)`
- `walk()` sets `_velocity` based on direction and `_walkSpeed`, sets `_actionState = ACTION_STATE_WALK`
- `ActionManager::update()` each frame: `_desiredPosition = position + velocity * dt`, then clamps to map bounds, sets position, reorders child Z-order by Y position
- Tower collision: Player characters are pushed around tower bounding boxes
- Speed: base 224, boostable to 320 via gear00

**Action states (enum actionState):**

```
ACTION_STATE_DEAD, IDLE, WALK, ATTACK, SATTACK, OATTACK, O2ATTACK,
HURT, ABHURT, KOCKDOWN, FLOAT, JUMP, AIRHURT
```

**Attack system:**

- `attack(abType type)` -- Triggers attack animation based on type (NAttack, SKILL1-3, OUGIS1-2)
- Attack animations are built from XML frame data with embedded commands: `setAttackBox`, `setSound`, `setMove`, `setJump`, `setCharge`, `setClone`, `setMon`, `setBullet`, `setTrap`, `setBuff`, `setCommand`, `setTransform`, `setOugis`, etc.
- `setAttackBox()` is the critical callback: sets `_effectType` and broadcasts `"acceptAttack"` notification via `CCNotificationCenter`
- All entities listen for `"acceptAttack"` and call `acceptAttack()` which handles collision detection

**Damage calculation (setDamage):**

- Base damage = `attackValue - target's _gardValue + random critical (0-49)`
- Armor reduction with `isHujia` (25% reduction), character-specific reductions
- Damage applied to `_hp` string, HP bar updated
- CKR (chakra/ultimate gauge) gained from taking/dealing damage
- CKR thresholds: 15000 for Ougis1, 25000 for Ougis2

**Hit types** (encoded in `_effectType`): `n_hit` (normal), `l_hit` (light knockback), `c_hit` (catch/stun), `o_hit` (ougi knockback), `o2_hit` (strong ougi), `f_hit` (float), `f2_hit` (high float), `b_hit` (bounce), `ab_hit` (absorb), `s_hit` (slow absorb), `ac_hit` (air catch), `ts_hit` (teleport catch), `sl_hit` (slow), `ct_hit` (catch/stick), `bc_hit` (bounce catch), `bf_hit` (bounce float), `ob_hit` (ougi bounce), `a_hit` (area).

## 7. INPUT HANDLING

**JoyStick** (virtual joystick):

- Extends `CCTouchDelegate`
- `ccTouchBegan/Moved/Ended` -- Detects touch, calculates direction vector from center
- Calls `_delegate->JoyStickUpdate(direction)` on move, `_delegate->JoyStickRelease()` on end
- `GameLayer` forwards these to `currentPlayer->walk(direction)` or `currentPlayer->idle()`

**ActionButton** (skill buttons):

- Each button has an `abType` (NAttack, SKILL1-3, OUGIS1-2, Item1, GearItem, GearBtn)
- `ccTouchBegan` -> `click()` -> calls `_delegate->attackButtonClick(_abType)`
- `ccTouchEnded` -> calls `_delegate->attackButtonRelease()`
- Cooldown system: `_coldDown` timer, progress bar animation, `isCanClick()` check
- Double-tap detection for certain skills (`_isDoubleSkill`, `_clickNum`, `_clickTime`)

**Gear buttons** -- Similar touch handling, delegates to `gearButtonClick(gearType)`

## 8. AI SYSTEM

AI is implemented per-character in `Characters.cpp`. Each character has a dedicated `AI_[Name]()` method. The AI runs on a timer (`doAI()` scheduled every ~0.5s).

**Common AI pattern:**

1. `findEnemy("Hero"/"Flog"/"Tower", range)` -- Searches for nearest enemy, sets `_mainTarget`
2. Calculate distance vector `sp = target.pos - self.pos`
3. Decision tree based on distance:
   - Far (>128-256px): Walk toward target
   - Medium (32-128px): Walk or use ranged skills
   - Close (<32px): Use melee attacks or close-range skills
4. Skill priority: OUGIS2 > OUGIS1 > SKILL3 > SKILL2 > SKILL1 > NAttack
5. Retreat logic: `checkRetri()` checks HP, `stepBack()` moves toward own base
6. Base defense: If `isBaseDanger`, AI prioritizes returning to defend
7. Gear purchasing: AI auto-buys gears when coins reach 500
8. Combat point awareness: `enemyCombatPoint` vs `friendCombatPoint` comparison for tactical retreat

**Special AI behaviors:**

- Summoned creatures (Karasu, Saso, Parents, Sanshouuo, Akamaru) follow their `_master` and attack nearby enemies
- Clones (Naruto clones) aggressively push lanes
- Guardian NPCs patrol and defend specific areas

## 9. MAP/LEVEL SYSTEM

**5 TMX tile maps** (`Tiles/1.tmx` through `5.tmx`), selected randomly in HardCore mode or default map 1 in Training.

**Map structure:**

- Object layer "object" -- Hero spawn points (6 positions for 3v3)
- Object layer "meta" -- Tower positions and dimensions
- Map width: 96 tiles * 32px = 3072px
- Two lanes: Konoha side (left, x~~13*32) and Akatsuki side (right, x~~83*32)

**Game entities placed on map:**

- 6 heroes (3 per team) at spawn points from TMX data
- 6 towers (3 per team) at meta positions
- 6 creeps per team per wave, spawned every 15 seconds
- Creeps walk along lanes toward enemy base

**Win condition:** Destroy all 3 of the enemy team's towers (or all towers on one side = 0).

## 10. GAME STATE MANAGEMENT

**Global state** (in `Defines.h`):

- `Cheats` -- Cheat code flag (affects game mode, 4v4 vs 3v3)
- `MemberID`, `GroupID`, `PWord` -- Player account info
- `isPlayed` -- Whether a game has been played
- `CURRENT_VERSION` -- Build version (12200)
- `SERVER` -- HTTP server URL for online features

**Per-game state** (in `GameLayer`):

- `currentPlayer` -- The human-controlled Hero
- `Heros` -- CCArray of hero data dictionaries (from selection)
- `_CharacterArray` -- CCArray of all active ActionManager entities
- `_TowerArray`, `_KonohaFlogArray`, `_AkatsukiFlogArray` -- Entity arrays
- `_minute`, `_second` -- Game clock
- `zhenying` -- Player's faction (0=Akatsuki, 1=Konoha)
- `_isHardCoreGame`, `_isOugis2Game`, `_isRandomChar` -- Game mode flags
- `_isSurrender`, `_isExiting` -- Exit state flags

**Per-entity state** (in `ActionManager`):

- `_hp`, `_maxHP` (CCString), `_ckr`, `_ckr2` -- Health and chakra
- `_level` (1-6), `_exp`, `_coin` -- Progression
- `_walkSpeed`, `_attackValue`, skill values/ranges/cooldowns
- `_isAI`, `_isControled`, `_isWudi` (invincible), `_isBati` (super armor), `_isBati`
- `_gearArray` -- Purchased items
- Buff system: `_healBuffValue`, `_powerUPBuffValue`, `_skillUPBuffValue`, `_dehealBuffValue`
- `_slayer` -- Who last attacked this entity (for kill credit)
- `_mainTarget` -- AI's current target

**Data persistence:** SQLite database via `KTools` for saving game records, character unlocks, skill configurations. MD5 checksums for file integrity verification.

## 11. NETWORKING CODE

**There is NO real-time multiplayer networking.** All "Network" features are HTTP-based REST calls for asynchronous data:

- **Login/Account:** `StartMenu` sends HTTP GET to `SERVER"nsk/list.php"` with player ID/password for login, profile, group management
- **Hero list loading:** `NetworkLayer` and `LoadLayer` fetch hero XML data from server
- **Score upload:** `GameOver` uploads match results (kills, time, character) to server
- **Rankings:** `RankingLayer` fetches ranking data (daily/weekly/monthly) from server
- **Skill unlock:** `SkillLayer` sends skill unlock requests to server

All networking uses `cocos2d::extension::CCHttpClient` (HTTP GET requests only), with async callbacks. The `NetworkLayer` class name is misleading -- it is actually the "Arcade mode" hero selection screen, not a networking layer.

**Server endpoint:** `http://localhost/` (defined in `Defines.h` as `SERVER`)

## KEY ARCHITECTURAL OBSERVATIONS

1. **Single-player with AI only** -- All 7 non-player characters are AI-controlled. The "Network" mode is just a different hero selection screen with team composition options.
2. **Data-driven character system** -- Character stats, animations, and skill data are loaded from XML files at runtime, making it relatively easy to add new characters.
3. **Notification-based combat** -- The `"acceptAttack"` notification broadcast system means any entity can hit any other entity without direct references, but it is also why combat is purely local.
4. **String-heavy architecture** -- Character names, group names, roles, attack types, and even HP values use CCString extensively rather than enums or numeric IDs, which would make networking synchronization very challenging.
5. **No game state serialization** -- Game state is entirely in-memory with no serialization/deserialization capability, which would be a prerequisite for any multiplayer implementation.
6. **Monolithic ActionManager** -- The 8000+ line ActionManager.cpp handles movement, combat, damage, buffs, items, skills, AI helpers, and visual effects all in one class, making it the most complex file in the codebase.
